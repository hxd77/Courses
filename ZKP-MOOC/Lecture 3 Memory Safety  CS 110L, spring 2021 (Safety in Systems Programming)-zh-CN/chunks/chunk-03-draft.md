## 把前/后置条件的思想搬到手动内存管理

石匠 Karel 问题告诉我们：只要把程序拆成一个个带有明确前置条件和后置条件的小函数，写程序的人就不必把整份代码一次性塞进脑子里，而只需要在每个函数内部关心自己那一小块。大图景自然会由这些零件拼起来。

但这里有一个关键前提：**这些前/后置条件完全由程序员自己负责维护**。函数签名里并不会写"这里你必须面朝南"，一般只能在注释里写一句"post condition: Karel 面朝东"。

这套思路完全适用于 CS 107 级别的内容——也就是你第一次接触**手动内存管理**的时候。当你在实现一个向量之类的数据结构时，真正需要在脑子里反复推演的问题是：

- 在这个程序里，我到底是怎么管理内存的？
- 大图景是什么？
- 一段内存会在什么地方被分配？又会被谁、在哪里释放？

任何足够复杂的程序都会在许多地方分配内存、把指针到处传递。如果某块内存释放得太早，别处可能还持有指向它的指针，于是就会出现**释放后使用（use-after-free）**；反过来，如果根本没人释放它，那就是**内存泄漏（memory leak）**。

所以你必须在**整体层面**明确：代码的哪些部分负责管理哪些内存？具体是哪个函数负责释放？释放的方式是什么？

## C/C++ 中"所有权"的惯例表达

如果你读过一些高质量的 C/C++ 代码，就会发现**所有权（ownership）**这个概念其实早已在工程界自发出现。过去三四十年里，程序员需要表达"这块内存该怎么用、由谁负责"，于是自然而然把"所有权"这个词用起来了。它在 C 语言本身里没有任何正式定义，但大家都在用它来给**手动内存管理**写一套非正式的前置条件和后置条件。

下面看几个真实项目里的例子。

### 真实项目里的例子

**例 1：openvswitch（来自 Stanford）**

这是 Stanford 出品的一个项目。在 GitHub 上搜一下就能看到类似下面的函数：这个函数分配了一段字符串，并把指向它的指针存进 `error_p`。函数的注释里写着：

> *"the caller is responsible for freeing `error_p`"*
> （调用者负责释放 `error_p`）

这就是一条**后置条件**：我帮你分配了内存，但你必须 call `free` 把它释放掉。

**例 2：ffmpeg**

ffmpeg 里有另一类约定。某个函数接收一个字典作为参数，注释里写着：

> *"the caller still owns `val` ... and is responsible for freeing it"*
> （调用者仍然拥有 `val`，并且负责释放它）

含义是：你把一个指针传给我，我只是临时看一看、顺手做点事情，但这块内存的所有权还在你手上，我不会去释放它。所以请你自己记得释放。

**例 3：把内存交出去给别人管**

还有一种更微妙的情况。某个函数的注释大致说：你传给我一段内存，**之后不用再操心它的释放**；我也不会在这里释放。因为我会把它转交给代码库里另一个组件——比如 `boot_sysfs_lib`——由那个组件在合适的时候释放。

在这里，所有权被一路交接：从调用者 → 当前函数 → `boot_sysfs_lib`。最终释放的是最后那个组件，它现在**拥有**这块内存，负责保证它最终会被清理掉。

### 自定义清理函数

上面几个例子里，内存最终都靠 `free` 来释放——这是你在 CS 107 里见得最多的情形。但在真实 C/C++ 代码中，许多数据结构有**自己专门的清理函数**。我们前面讨论向量时就提到过 `vec_free`。

为什么需要自定义清理函数？因为 C 里很常见这样的场景：一个结构体里有指针指向另一个结构体，后者里又有指针指向下一个结构体……如果你只对最外层的结构体调用 `free`，`free` 根本意识不到里面还有别的指针需要先释放。于是你必须写一个专用的清理函数，让它按顺序：

1. 沿着指针链走进去，释放最里层指向的东西；
2. 逐级往外释放；
3. 最后释放最外层的结构体。

所以你会看到这样的注释：

> *"the caller owns the data in `port`"*
> （调用者拥有 `port` 中的数据）

意思是调用者**负责释放**这块内存，但**不能**直接对它调用 `free`；必须调用那个自定义的清理函数——它会沿着一串指针把各种关联信息一并释放。如果在这种情况下你错误地直接 `free`，就会造成内存泄漏：因为你只释放了最外层的结构体，里面那些被指向的内存还原封不动地留在堆上。

类似的模式在别处也反复出现：要释放某个对象，必须调用这个特定的"释放函数"，而不是标准的 `free`。

### 键和值分别归属

情况还能更复杂。下面这个例子里，函数拿到一个字典，而规则是：

- 代码库的**某一部分**负责释放字典的**键（keys）**；
- **另一部分**负责释放字典的**值（values）**。

为什么会这样？想象一种使用场景：一段时间过后，我们不再需要这些值了，只关心键；又或者正相反，我们不再关心键，只想继续用这些值。于是一方先释放自己负责的那一半，剩下那一半由另一方在它完成自己的工作后再释放。

这种所有权在同一个数据结构里**按字段分裂**的情况，推理起来非常吃力，但它是真实世界里的正常代码。手动内存管理的本质就是这样——它不总是简单的。你必须**明确**：谁负责做什么？

所以 C/C++ 代码里自然而然地形成了一种**原始形式的所有权概念**：

- 这块内存在这里被分配；
- 然后被传递给代码库的另一个组件，从此那个组件负责释放；
- 或者：我分配了内存，只是临时借给某个辅助函数用一下；辅助函数做完事以后，**所有权仍然在我这里**，还是我来负责释放。

许多程序员喜欢**把代码拟人化**地看待："我是一个函数，我要把这个值交给另一个人（另一个函数）。"所有权模型和这种拟人化思考方式是天然契合的：你把自己想成代码库里的某一块，然后问——此刻我关心什么？这块内存归我吗？我是要自己清理，还是要把它交给别人去清理？

## C 类型系统的局限

问题在于：以上这些**所有权约定**，在 C 代码里只能写在**注释**里。你可以在函数文档里写"本函数接收一个缓冲区，但不会释放它，所有权仍在你手上，你必须自己 `free`"；调用方只能**读注释**才能知道这件事。

- 在 CS 106A 里，犯错就是程序崩溃，代价有限。
- 在 CS 107 里，如果你的内存管理出错，程序不一定会崩——它可能看起来运行得完美无缺。但一旦**攻击者**发现了这个漏洞，后果可以从内存泄漏（泄漏得够多也会耗尽内存、最终崩溃），一路到**双重释放（double free）**、**释放后使用（use-after-free）**、**缓冲区溢出（buffer overflow）**乃至**远程代码执行（RCE）**——最坏情况下攻击者直接接管你的程序。

而**编译器根本帮不上忙**。前几次课讲过静态分析，第一周的练习也会让你看到它的严重局限。

这一段内容里我希望你记住的**核心一句话**是：

> **编译器不知道你的前/后置条件是什么，因为 C 语言里没有任何语法能表达它们。**

你可以在注释里写"调用者负责释放"、"所有权已转移"、"这块内存会交给另一个函数释放"，但编译器不会读英语注释，C 语言本身也没有任何关键字去说"这个函数负责释放这块内存"或"这个函数会把这块内存交给另一个函数去释放"。对编译器来说，这些话根本不存在。

问题的根源在于**语言的类型系统（type system）**。打个比方：一门编程语言里的**类型**，就像一门自然语言里的**名词**——你说话的时候总要谈论某些"东西"，在编程里你谈论的就是类型。C 的类型系统里有的"名词"非常少，基本只有：

- 数值类型（数字）；
- 指针；
- 结构体；
- （再加上）函数。

当你写 C 程序时，你能谈论的就是这些东西：函数接收一个数，返回一个指针，分配一块内存——差不多就这样了。整门 C 语言，在 CS 107 学半个季度基本就能学完。

**简单（simple）不等于容易（easy）**。你脑子里的想法可能很复杂——"我有一张列表，列表里的元素有若干特定属性"——但你必须把这些复杂的想法翻译成一门"名词"极少的原始语言，这就是 C 类型系统的本质。

### strdup 的启示

正因为 C 的类型系统这么小，才会出现 `strdup` 这类函数带来的问题。`strdup` 的 C 签名大致是这样：

```c
char *strdup(const char *s);
```

从编译器的视角看，这个函数不过是：接收一个指向字符序列的指针，返回一个指向字符序列的指针——它甚至不知道"字符串"这个概念，只看到 `char*`。

于是你可以这样误用它：

```c
char *duplicate = strdup(some_string);
return duplicate;   // 函数在这里返回，没有释放 duplicate
```

编译器分析这段代码，会觉得一切正常：你传给 `strdup` 一个 `char*`，拿到一个 `char*`，又把它返回——类型完全匹配，挑不出毛病。

但是如果你读过 `strdup` 的 man page，会看到它写着：

> *"allocates memory for a new string with `malloc`; the memory should be freed with `free`."*
> （用 `malloc` 为新字符串分配内存；该内存应当用 `free` 释放。）

一个有经验的程序员看到上面那段代码就会马上意识到：`strdup` 返回的是一段**堆分配（heap allocated）**的字符串，但我们没有 `free` 就直接 `return` 了——这就是一处内存泄漏。

然而编译器看不到这些。编译器不读 man page，也不理解注释里那段自由英文。它**不知道这里存在一条后置条件**：「返回的是堆内存，必须由某人去释放」。

结论再强调一遍：你可以把前置条件和后置条件写进注释，但如果语言本身没有提供一种方式把它们**写进代码里**，编译器就无从得知你的意图。它能验证最基础的类型——你接收了一个数，你返回了一个布尔值，你返回了一个指向字符序列的指针——仅此而已。再往上一层的**意图**，它不懂，因此当你做错时，它也不会报错。

### 一个更好的类型系统？

这就带出本课的**核心问题**：

> **能不能设计一种更强的类型系统，让我们能把意图直接写在代码里，并由编译器去检查？**

如果编译器真的能理解你的前/后置条件——比如它能理解"这个函数返回一段堆分配内存，必须由某人调用 `free` 释放它"——那么它就可以反过来去检查：这块内存确实在某处被释放了吗？一旦能做到这一点，编译器就能替你核对你有没有遵守自己承诺的前置条件和后置条件，你的代码质量会因此大大提升。
