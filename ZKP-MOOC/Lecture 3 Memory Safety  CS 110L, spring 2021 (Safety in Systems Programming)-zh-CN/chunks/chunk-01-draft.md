这段代码里问题很多，**而且有一类很有意思的现象**：有些地方你一眼看过去觉得"这肯定不对"，但因为别处还有更严重的 bug 盖过了它，它反而算不上主要问题。我自己数下来至少有 7 处错误，之后我会把完整清单贴到 Slack 上。

举几个典型的例子：

- **向量初始容量是 0**：`vec_push` 在需要扩容时，是把当前容量乘以 2。问题是 0 乘以 2 还是 0，所以 `vec_push` 永远触发不了正常的扩容路径，直接出错。如果一开始就把向量初始化成一个合理的容量，这个问题根本就不会出现。
- **`vec_push` 在扩容时没释放旧缓冲区**：这就是下面要重点讲的内存泄漏。

> **向量扩容背景补充**：对不熟悉 vector 实现的同学解释一下——向量的典型做法是先分配一个固定大小的数组（缓冲区）；当容量不够时，丢弃当前数组，分配一个**两倍大**的新数组，把原有元素拷过去。这是绝大多数 vector 实现的套路。

接下来我们挑其中**最重要的四类 bug**过一遍。

### 内存泄漏（Memory Leak）

先说最容易理解的：**内存泄漏**。

刚才提到，`vec_push` 在扩容时会分配一块**两倍大**的新缓冲区（就是示意图里的 `new_data`），把旧缓冲区的数据拷过去，然后让 `vec->data` 指向新缓冲区。

用文字描述一下示意图上发生的事：

1. 最初，向量的 `vec->data` 指针指向一段旧缓冲区，该缓冲区已经装满。
2. 我们要 push 一个新元素，于是分配一段容量翻倍的新缓冲区。
3. 把旧缓冲区里的元素逐个拷贝到新缓冲区。
4. 让 `vec->data` 指向新缓冲区，之后的操作都在新缓冲区上进行。
5. **但旧缓冲区**——那块谁也不再引用的内存——**我们忘了 free**。它就这样一直留在堆里，没人用也没人释放。这就是典型的**内存泄漏（memory leak）**。

（*不过关于这里到底算不算"内存泄漏"，还有点哲学上的讨论空间*：如果去看 `vec_new` 的实现，你会发现一开始根本就没分配初始数组，`vec->data` 直接是 `NULL`。那么"丢弃一个原本就不存在的分配"还算不算泄漏呢？——这个问题就留给大家争论了。总之，这段代码 bug 太多，互相纠缠，以至于"到底哪里该改、怎么改"本身就很难分析清楚。）

### 双重释放（Double Free）与远程代码执行

第二类问题是**双重释放（double free）**。

看 `main` 函数最后两行的做法：它**先对 `vec->data` 调用一次 `free`**，**紧接着又调用 `vec_free`**。而如果你去看 `vec_free` 的实现，里面**还会再对 `vec->data` 调用一次 `free`**。结果就是，同一块缓冲区被释放了两次。

乍一看，双重释放好像没什么大不了的：不就是多调了一次 `free` 嘛，能出什么事？

——但**实际上这是一个非常严重的 bug**，甚至可以直接导致**远程代码执行（remote code execution，RCE）**。所谓远程代码执行，是指攻击者给你的程序发送恶意数据后，就能夺取对你二进制程序的控制权：本该在二进制的代码段（text segment）里跑的指令，被攻击者发来的指令取代——这是最严重的一类安全问题。

我当初第一次看到"double free 能导致 RCE"时也完全不敢相信，因为这代码看起来实在太无辜了。如果你上过 **CS 155**（Stanford 的计算机安全课），里面有一道作业就是让你**从一次 double free 一路拿到 root 权限**：题目给你一段带 double free 的代码，要求你"拿到 root 权限"。一开始我觉得"这不可能吧"，结果做完整套作业后，你真的会惊呼——"哇，我真的拿到 root 了"。
