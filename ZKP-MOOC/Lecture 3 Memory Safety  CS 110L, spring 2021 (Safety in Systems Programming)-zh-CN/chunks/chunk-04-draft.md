## 进入 Rust：所有权模型

之前的讨论已经展示了"所有权"可以存在于注释或约定里，但语言本身并不会强迫你按约定走——没有什么机制能强制你释放内存。接下来要问的问题就是：**如果"所有权"这件事真的被写进编程语言里，会是什么样？** 下面先给出一个直观的 Rust 入门介绍，周四会再做深入讲解。

### 一行 Rust 代码与"谁拥有小熊"

看一下这一行 Rust：

```rust
let julio = bear_get();
```

形式上它跟你见过的其他语言差不多：声明一个变量 `julio`，让它等于某个函数的返回值。你可以想象 `bear_get()` 交给你的是一只小熊玩偶。

在 Rust 里需要记住的第一件事是：**每一个变量都"拥有"它持有的那个值**。这行代码里，`julio` 就是这只小熊的所有者（owner）——这只熊属于我，不属于别人，我可以对它做任何事。而既然我是它的所有者，**清理这只熊的责任也落在我身上**：玩完之后得把它放回去。

可以把场景想成一家日托所，架子上摆着一堆玩具。你从玩具架上拿一只玩具玩，玩爽了、玩够了，最后得把它放回你拿它的那个架子——这里对应的就是"玩具架"。

放到代码里就是这样一个场景：在 `main` 函数里写 `let julio = bear_get();`，当这个函数返回时，**是 `julio` 这个所有者——也就是"我"——负责把小熊放回去，没人会替我做**。

可以这样小结一下：等号左边的变量名（这里是 `julio`）是等号右边那个对象的所有者。我可以对这个玩具做任何事——按一下它身上的按钮让它发光，或者调用它的其他方法。但每次都要回到同一件事上：**把小熊放回去、释放它的内存，是所有者的责任。**

### 赋值触发所有权移动

给程序再加一行，情况就变得有意思了：

```rust
let julio = bear_get();
let ryan = julio;
```

一开始小熊在我（`julio`）手里。但当 `let ryan = julio;` 被执行，你应该在脑海里看到这样一幕：**小熊从 Julio 手里移动（move）到了 Ryan 手里**。

于是现在 Ryan 才是这只玩具的所有者。原来 Julio 能对小熊做的事——按按钮、让它亮起来、调用任何方法——**现在只有 Ryan 能做**。与此对应，**把小熊放回玩具架、释放这块内存的责任，也完全转到了 Ryan 身上**，和 Julio 没有关系了。

那问题来了：`julio` 这个变量现在会怎样？聊天区已经有同学答对了：**"Julio 什么都干不成了，这对我来说就结束了。"** 确实如此。从 `julio` 的视角看，小熊已经不在我这儿了——我把它交给 Ryan 了。我**没有所有权**，再去碰它只会出问题。在 Rust 里，任何尝试使用已经移出的 `julio` 的行为都会被编译器拦下。

不过也不全是坏消息：**我不再需要负责释放这只熊了**——不用把它放回架子上，也不用释放那块内存。这件事现在是 Ryan 的事，我可以安心走开。

你已经开始看出这个模型怎么回避之前讨论的那些内存问题了：现在"谁来释放"这件事被**显式地委托**给了某个唯一的所有者，一旦值交出去，责任就跟着走，不会有两个人同时以为该释放它。

### 一个看得见的例子

把上面的小熊换成一个字符串，跑一个真实的 Rust 例子：

```rust
let julio = String::from("hi friends");
let ryan = julio;
println!("{}", ryan);
```

右边可以是任何对象，这里具体是一个 `"hi friends"` 字符串。紧接着写 `let ryan = julio;`——和前面一样，这只"熊"（字符串）现在属于 Ryan 了。最后用 `println!` 打印（`println!` 加上感叹号，这是 Rust 打印的写法）。这段代码运行时一切正常：Ryan 是所有者，所以能打印出 `hi friends`。

但如果紧接着再试一次打印 `julio` 呢？

```rust
println!("{}", ryan);
println!("{}", julio); // 编译错误
```

如果换成别的语言，你可能以为这会输出两次 `hi friends`——毕竟 `julio` 和 `ryan` "看起来"指向同一个东西。但在 Rust 里，编译器能识别出：**这只小熊、这个字符串，已经不再属于 `julio` 这个变量了。** 编译器会给出一段信息量很大的报错，其中关键一句是：

```
error[E0382]: borrow of moved value: `julio`
```

翻译过来就是：`julio` 里原来持有的值已经被移到别处去了，而你又想再用这个变量——可是你手里已经什么都没有了，它不再归你。

换一种直观说法：`julio` 这个变量名"指向"它拥有的对象，而 `println!` 想做的是打印 `julio` 所拥有的东西。可到这一步 `julio` 什么都没拥有，所以只有 `ryan` 作为所有者时才能打印。

> **学生**：你是在说编译器那段 "borrow of moved value" 的报错措辞吗？
>
> **讲者**：是的。编译器其实没法"真的知道"你心里在想什么，这只是它在当前上下文里能给出的最准确的描述——"你原来放在 `julio` 里的那个东西，已经被移走了"。

换个比喻，你可以想象 **Rust 编译器就站在旁边、眼睛一直盯着：谁在拥有什么？什么时候被移走？** 一旦 `julio` 里的内容被移到 `ryan`，之后任何地方再出现 `julio` 这个变量名，编译器都会立刻反应过来："这个值已经不在了。" 正因为它盯得这么紧，才能给出很友好的报错信息，让这些问题更容易被发现。

### 把 `julio` 重新赋给别的东西

有同学问：那是不是 `julio` 从此就废了？并不是。**值被移走之后，你仍然可以把 `julio` 赋给另一个新的对象。** 比如现在 `ryan` 手里拿着那只熊，我可以再拿一只新的熊，让 `julio` 拥有它，这完全没问题——只要**每一个值都只有一个所有者**，这条规则不被破坏就行。

（聊天区问到："Rust 里变量不是默认不可变吗？" 是个很好的问题。关于不可变性和 `mut` 的规则，周四会专门讲，今天先从宏观上把所有权模型理解清楚。）

## 函数调用与所有权

上面看到的是**变量之间赋值**会触发所有权移动。还有一种常见情形也会触发同样的移动——**函数调用**。

```rust
let julio = bear_get();
my_cool_bear_function(julio);
```

这里把 `julio` 作为参数传给了 `my_cool_bear_function`。**这个函数就把小熊拿走了**——和前面 Ryan 从我手里拿走熊是一回事。于是调用之后：

- 在 `main` 里再试图用 `julio` 做任何事（比如打印）都会失败；
- **释放这只熊的责任现在转给了这个函数**，由它决定什么时候放回去；调用者 `main` 已经不再负责了。

## 借用：把分解找回来

这时马上会出现一个担忧：**那以后还怎么做代码分解（decomposition）？** Ryan 前面刚提过，CS 106A 头两周就在教分解；可现在又告诉你：把变量传进函数，就等于失去它的所有权，传完再也用不了——这几乎让任何想做的分解都没法做。

好在 Rust 还有另一个特性叫**借用（borrowing）**。直观想一下：我持有这个玩具，我愿意把它**借给**别人用一小会儿——让他们打印它一下，或者做点别的——然后他们**用完得还给我**，因为它本来就是我的。

写成代码就是加一个 `&`：

```rust
let julio = bear_get();
my_cool_bear_function(&julio);
```

如果你有 C 背景，看到 `&`（ampersand）可能会犯嘀咕："我是在传地址吗？" 从底层看它确实是引用。但更贴切的理解是：**我允许这个函数借用我的小熊、借用我的字符串、借用我的对象；函数返回之后，这只熊还在我手里**，我可以继续像以前一样写代码，根本不用担心变量"被移走了"。

简而言之：**借用出去了要记得还，释放的责任仍然在原来的所有者身上，只是有那么一小会儿这个东西在别人手上。**

讲者举了个更生活化的类比：**就像我把任天堂 DS 借给朋友玩**——理论上他会还给我，然后 DS 还在我这儿。Rust 里的借用就是这样。

有了 `&` 这个借用机制，**你熟悉和喜欢的那种代码分解就回来了**：函数可以操作值，但不必吃掉值。

## 小结

把上面的内容拉到宏观层面，Rust 官方教程（Rust Book，课件里有链接，写得非常清楚）对所有权的总结基本就是这样几条：

- **每一个值，在任意时刻只能有一个所有者。**
- **当这个所有者离开作用域时，值会被释放。** 比如前面 `main` 函数走到右花括号 `}` 的那一刻，Rust 就会替你把那时该释放的东西释放掉。你可以想象所有释放动作都是在这些花括号处、由编译器按"谁是所有者"这条线索自动安排好的。
- **谁是所有者，谁就是释放的责任人**——这正好避免了前面讨论的那种双重释放（double free），因为不会再出现两边都以为自己该释放的情形。
- **最关键的一点：所有这些检查都发生在编译期（compile time）。** 程序根本没跑起来，就像刚才那个字符串例子一样——编译器直接拦住你："你违反了所有权规则，我不知道你想做什么。" 这就是 Rust 相比 C 的最大优势：C 可以让你在没有释放的情况下照样编译通过，因为这类检查根本没有内建在 C 编译器里；而 Rust 把这些检查内建进编译器，对内存安全问题帮助非常大。

下一次课我们会再讲 Rust 里其他类型的变量、这套系统还能回避哪些内存问题，并看更多代码示例。

### 课堂尾声问答

> **学生**（下课离开前的一个问题）：在 Rust 里，栈（stack）和堆（heap）还存在吗？
>
> **讲者**：当然还在。周四我们会更细地讲内存布局，但你可以这样想：刚才例子里的那只"熊"、或者那个字符串，是**在堆上**的；而等号左边的 `julio`、`ryan` 这些变量本身**在栈上**，相当于指向堆上对象的指针——和 C 的模型没什么两样，只是在这个模型之上多加了一层由编译器强制执行的规则。

所有权一开始会让人觉得有点别扭，头几次你肯定会跟编译器"打一阵架"，但一旦这个心智模型建立起来，后面的一切都会顺很多——因为它真正把"这块内存到底该由谁来释放"这件事说清楚了。
