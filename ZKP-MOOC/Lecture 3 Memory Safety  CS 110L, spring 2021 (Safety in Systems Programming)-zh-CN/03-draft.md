### 内存泄漏（Memory Leak）

先说最容易理解的：**内存泄漏**。

刚才提到，`vec_push` 在扩容时会分配一块**两倍大**的新缓冲区（就是示意图里的 `new_data`），把旧缓冲区的数据拷过去，然后让 `vec->data` 指向新缓冲区。

用文字描述一下示意图上发生的事：

1. 最初，向量的 `vec->data` 指针指向一段旧缓冲区，该缓冲区已经装满。
2. 我们要 push 一个新元素，于是分配一段容量翻倍的新缓冲区。
3. 把旧缓冲区里的元素逐个拷贝到新缓冲区。
4. 让 `vec->data` 指向新缓冲区，之后的操作都在新缓冲区上进行。
5. **但旧缓冲区**——那块谁也不再引用的内存——**我们忘了 free**。它就这样一直留在堆里，没人用也没人释放。这就是典型的**内存泄漏（memory leak）**。

（*不过关于这里到底算不算"内存泄漏"，还有点哲学上的讨论空间*：如果去看 `vec_new` 的实现，你会发现一开始根本就没分配初始数组，`vec->data` 直接是 `NULL`。那么"丢弃一个原本就不存在的分配"还算不算泄漏呢？——这个问题就留给大家争论了。总之，这段代码 bug 太多，互相纠缠，以至于"到底哪里该改、怎么改"本身就很难分析清楚。）

### 双重释放（Double Free）与远程代码执行

第二类问题是**双重释放（double free）**。

看 `main` 函数最后两行的做法：它**先对 `vec->data` 调用一次 `free`**，**紧接着又调用 `vec_free`**。而如果你去看 `vec_free` 的实现，里面**还会再对 `vec->data` 调用一次 `free`**。结果就是，同一块缓冲区被释放了两次。

乍一看，双重释放好像没什么大不了的：不就是多调了一次 `free` 嘛，能出什么事？

——但**实际上这是一个非常严重的 bug**，甚至可以直接导致**远程代码执行（remote code execution，RCE）**。所谓远程代码执行，是指攻击者给你的程序发送恶意数据后，就能夺取对你二进制程序的控制权：本该在二进制的代码段（text segment）里跑的指令，被攻击者发来的指令取代——这是最严重的一类安全问题。

我当初第一次看到"double free 能导致 RCE"时也完全不敢相信，因为这代码看起来实在太无辜了。如果你上过 **CS 155**（Stanford 的计算机安全课），里面有一道作业就是让你**从一次 double free 一路拿到 root 权限**：题目给你一段带 double free 的代码，要求你"拿到 root 权限"。一开始我觉得"这不可能吧"，结果做完整套作业后，你真的会惊呼——"哇，我真的拿到 root 了"。

### 迭代器失效（Iterator Invalidation）

这段代码里还藏着另一种形式的悬垂指针问题，叫作**迭代器失效（iterator invalidation）**。

迭代器（iterator）可以理解为指向容器内某个位置的引用。通常迭代器从容器的开头出发，你让它逐个元素地向后移动，遍历整个容器。但如果你在遍历过程中**修改了容器**——比如向 vector 里 push 新元素——迭代器就可能失效，按约定你不应该再使用它了。然而，C/C++ 的编译器并不会阻止你继续用它。

回到代码。这里的 `n` 就是一个迭代器，它指向向量开头的元素。示意图里：

- 向量底层有一块缓冲区（buffer），`n` 指向缓冲区的第一个元素。
- 当我们执行 `vec_push` 时，如果恰好触发了重新分配（reallocation），就会分配一块新的缓冲区，把原来的数据拷贝过去，`vec->data` 随之指向这块新缓冲区。
- 理想情况下，旧的缓冲区此时应该被释放——顺便说一句，前面代码里根本没释放它，所以其实还叠加了一个**内存泄漏（memory leak）**。换句话说，这个迭代器失效 bug 之所以不那么"立即致命"，是因为另一个 bug（内存泄漏）帮它挡了一下。但按理说旧缓冲区本来就该被释放。

一旦旧缓冲区被释放，`n` 就指向了一块已经释放的内存。已经释放的内存可能被其他代码重新拿去用了——也就是说，原来的元素可能已经不在那里了。如果你继续执行 `printf`，把 `n` 解引用打印出来，你就是在读一块任意数据，谁也不知道里面现在装的是什么，因为它早被释放、甚至可能被别的模块复用了。

### 从石匠 Karel 说起

我想暂时跳出内存这个话题，回到 **CS 106A**（Stanford 的入门编程课），讲讲**石匠 Karel 问题（Stonemason Karel）**。

对没上过 CS 106A 的同学简单交代一下背景：这是 CS 106A 第一周就会遇到的一道题。场景里有一个叫 Karel 的小机器人，它的任务是在一排**柱子**上摆放**铃铛（beepers，图中画成菱形）**。柱子之间彼此相隔——每两根柱子之间空三格，也就是**每隔四格立一根柱子**。Karel 需要爬上一根柱子，把这根柱子从下到上摆满铃铛，然后下来，走到下一根柱子，再爬上去逐格摆铃铛，再下来，再走到下一根……如此反复，直到走完。

这个问题在今天看来似乎很 trivial，但对初学者而言实现起来其实相当困难、也相当难以推理。因为你会发现：

- 要做的事情不少；
- 每一步做什么取决于当前所处的位置；
- 控制流（control flow）并不简单——怎么把它组织好？

初学者的典型反应是：**试图把整个解法一次性塞进脑子里**。

而我们在 CS 106A 前两周拼命想灌输给学生的核心思想就是：**分解，分解，分解（decompose, decompose, decompose）**。把问题拆成若干个可以**独立推理**的小组件，再把它们拼装成整体解法。

对石匠 Karel 问题来说，一个自然的分解就是：

> **while** 前方没有到墙（front is clear）：
>   &nbsp;&nbsp;修一根柱子（repair a column）；
>   &nbsp;&nbsp;移动到下一根柱子（move to next column）。

也就是先定下一个高层计划，把各个组件组装在一起；然后**单独实现每一个小模块，不必同时在脑中装下整个程序**。

回到我们那段 vector 代码——问题的本质也正在于此：**缺结构、缺规划**。你被迫同时想着所有事：

- "我现在在这里 `free`，可这个指针其实指向栈上的某个东西……"
- "我现在在这里 reallocate，那这里、这里、那里都得一起考虑……"

我们不想这样。我们希望能够**从高层去把握大图景**，同时**在实现每个小模块时只关注这个模块本身**，不必把整个程序都装进脑子。

### 前置条件与后置条件

我带过好几学期的 CS 106A，见过很多学生把"修一根柱子"这个函数写成：**有时候 Karel 结束时朝南，有时候朝东**。这就麻烦了。

因为一旦你的"移动到下一根柱子"函数假设"直接向前走四格"，而这时 Karel 恰巧朝南——那它就会往地里钻，试图向下走四格，结果一头撞到最底下的墙上，game over。

所以我们在 106A 里会反复强调**前置条件与后置条件（pre/post conditions）**：

- **前置条件**：函数开始运行之前必须满足的条件；
- **后置条件**：函数运行结束之后必须满足的条件。

比如"修一根柱子"的后置条件可以是"Karel 结束时朝南"，也可以是"Karel 结束时朝东"——朝哪边都行，**关键是它必须保持一致**。因为只有这样，"移动到下一根柱子"才能确定 Karel 的姿态，才能在开头放上合适的旋转或移动指令，让整件事跑通。

有了前置条件和后置条件，我们就能**孤立地推理每一个小模块**。只要能验证"前一个函数的后置条件"与"后一个函数的前置条件"吻合——也就是说，这些组件在接口上能严丝合缝地对接——我们就可以把它们串起来，确信整体也能跑通。
## 把前/后置条件的思想搬到手动内存管理

石匠 Karel 问题告诉我们：只要把程序拆成一个个带有明确前置条件和后置条件的小函数，写程序的人就不必把整份代码一次性塞进脑子里，而只需要在每个函数内部关心自己那一小块。大图景自然会由这些零件拼起来。

但这里有一个关键前提：**这些前/后置条件完全由程序员自己负责维护**。函数签名里并不会写"这里你必须面朝南"，一般只能在注释里写一句"post condition: Karel 面朝东"。

这套思路完全适用于 CS 107 级别的内容——也就是你第一次接触**手动内存管理**的时候。当你在实现一个向量之类的数据结构时，真正需要在脑子里反复推演的问题是：

- 在这个程序里，我到底是怎么管理内存的？
- 大图景是什么？
- 一段内存会在什么地方被分配？又会被谁、在哪里释放？

任何足够复杂的程序都会在许多地方分配内存、把指针到处传递。如果某块内存释放得太早，别处可能还持有指向它的指针，于是就会出现**释放后使用（use-after-free）**；反过来，如果根本没人释放它，那就是**内存泄漏（memory leak）**。

所以你必须在**整体层面**明确：代码的哪些部分负责管理哪些内存？具体是哪个函数负责释放？释放的方式是什么？

## C/C++ 中"所有权"的惯例表达

如果你读过一些高质量的 C/C++ 代码，就会发现**所有权（ownership）**这个概念其实早已在工程界自发出现。过去三四十年里，程序员需要表达"这块内存该怎么用、由谁负责"，于是自然而然把"所有权"这个词用起来了。它在 C 语言本身里没有任何正式定义，但大家都在用它来给**手动内存管理**写一套非正式的前置条件和后置条件。

下面看几个真实项目里的例子。

### 真实项目里的例子

**例 1：openvswitch（来自 Stanford）**

这是 Stanford 出品的一个项目。在 GitHub 上搜一下就能看到类似下面的函数：这个函数分配了一段字符串，并把指向它的指针存进 `error_p`。函数的注释里写着：

> *"the caller is responsible for freeing `error_p`"*
> （调用者负责释放 `error_p`）

这就是一条**后置条件**：我帮你分配了内存，但你必须 call `free` 把它释放掉。

**例 2：ffmpeg**

ffmpeg 里有另一类约定。某个函数接收一个字典作为参数，注释里写着：

> *"the caller still owns `val` ... and is responsible for freeing it"*
> （调用者仍然拥有 `val`，并且负责释放它）

含义是：你把一个指针传给我，我只是临时看一看、顺手做点事情，但这块内存的所有权还在你手上，我不会去释放它。所以请你自己记得释放。

**例 3：把内存交出去给别人管**

还有一种更微妙的情况。某个函数的注释大致说：你传给我一段内存，**之后不用再操心它的释放**；我也不会在这里释放。因为我会把它转交给代码库里另一个组件——比如 `boot_sysfs_lib`——由那个组件在合适的时候释放。

在这里，所有权被一路交接：从调用者 → 当前函数 → `boot_sysfs_lib`。最终释放的是最后那个组件，它现在**拥有**这块内存，负责保证它最终会被清理掉。

### 自定义清理函数

上面几个例子里，内存最终都靠 `free` 来释放——这是你在 CS 107 里见得最多的情形。但在真实 C/C++ 代码中，许多数据结构有**自己专门的清理函数**。我们前面讨论向量时就提到过 `vec_free`。

为什么需要自定义清理函数？因为 C 里很常见这样的场景：一个结构体里有指针指向另一个结构体，后者里又有指针指向下一个结构体……如果你只对最外层的结构体调用 `free`，`free` 根本意识不到里面还有别的指针需要先释放。于是你必须写一个专用的清理函数，让它按顺序：

1. 沿着指针链走进去，释放最里层指向的东西；
2. 逐级往外释放；
3. 最后释放最外层的结构体。

所以你会看到这样的注释：

> *"the caller owns the data in `port`"*
> （调用者拥有 `port` 中的数据）

意思是调用者**负责释放**这块内存，但**不能**直接对它调用 `free`；必须调用那个自定义的清理函数——它会沿着一串指针把各种关联信息一并释放。如果在这种情况下你错误地直接 `free`，就会造成内存泄漏：因为你只释放了最外层的结构体，里面那些被指向的内存还原封不动地留在堆上。

类似的模式在别处也反复出现：要释放某个对象，必须调用这个特定的"释放函数"，而不是标准的 `free`。

### 键和值分别归属

情况还能更复杂。下面这个例子里，函数拿到一个字典，而规则是：

- 代码库的**某一部分**负责释放字典的**键（keys）**；
- **另一部分**负责释放字典的**值（values）**。

为什么会这样？想象一种使用场景：一段时间过后，我们不再需要这些值了，只关心键；又或者正相反，我们不再关心键，只想继续用这些值。于是一方先释放自己负责的那一半，剩下那一半由另一方在它完成自己的工作后再释放。

这种所有权在同一个数据结构里**按字段分裂**的情况，推理起来非常吃力，但它是真实世界里的正常代码。手动内存管理的本质就是这样——它不总是简单的。你必须**明确**：谁负责做什么？

所以 C/C++ 代码里自然而然地形成了一种**原始形式的所有权概念**：

- 这块内存在这里被分配；
- 然后被传递给代码库的另一个组件，从此那个组件负责释放；
- 或者：我分配了内存，只是临时借给某个辅助函数用一下；辅助函数做完事以后，**所有权仍然在我这里**，还是我来负责释放。

许多程序员喜欢**把代码拟人化**地看待："我是一个函数，我要把这个值交给另一个人（另一个函数）。"所有权模型和这种拟人化思考方式是天然契合的：你把自己想成代码库里的某一块，然后问——此刻我关心什么？这块内存归我吗？我是要自己清理，还是要把它交给别人去清理？

## C 类型系统的局限

问题在于：以上这些**所有权约定**，在 C 代码里只能写在**注释**里。你可以在函数文档里写"本函数接收一个缓冲区，但不会释放它，所有权仍在你手上，你必须自己 `free`"；调用方只能**读注释**才能知道这件事。

- 在 CS 106A 里，犯错就是程序崩溃，代价有限。
- 在 CS 107 里，如果你的内存管理出错，程序不一定会崩——它可能看起来运行得完美无缺。但一旦**攻击者**发现了这个漏洞，后果可以从内存泄漏（泄漏得够多也会耗尽内存、最终崩溃），一路到**双重释放（double free）**、**释放后使用（use-after-free）**、**缓冲区溢出（buffer overflow）**乃至**远程代码执行（RCE）**——最坏情况下攻击者直接接管你的程序。

而**编译器根本帮不上忙**。前几次课讲过静态分析，第一周的练习也会让你看到它的严重局限。

这一段内容里我希望你记住的**核心一句话**是：

> **编译器不知道你的前/后置条件是什么，因为 C 语言里没有任何语法能表达它们。**

你可以在注释里写"调用者负责释放"、"所有权已转移"、"这块内存会交给另一个函数释放"，但编译器不会读英语注释，C 语言本身也没有任何关键字去说"这个函数负责释放这块内存"或"这个函数会把这块内存交给另一个函数去释放"。对编译器来说，这些话根本不存在。

问题的根源在于**语言的类型系统（type system）**。打个比方：一门编程语言里的**类型**，就像一门自然语言里的**名词**——你说话的时候总要谈论某些"东西"，在编程里你谈论的就是类型。C 的类型系统里有的"名词"非常少，基本只有：

- 数值类型（数字）；
- 指针；
- 结构体；
- （再加上）函数。

当你写 C 程序时，你能谈论的就是这些东西：函数接收一个数，返回一个指针，分配一块内存——差不多就这样了。整门 C 语言，在 CS 107 学半个季度基本就能学完。

**简单（simple）不等于容易（easy）**。你脑子里的想法可能很复杂——"我有一张列表，列表里的元素有若干特定属性"——但你必须把这些复杂的想法翻译成一门"名词"极少的原始语言，这就是 C 类型系统的本质。

### strdup 的启示

正因为 C 的类型系统这么小，才会出现 `strdup` 这类函数带来的问题。`strdup` 的 C 签名大致是这样：

```c
char *strdup(const char *s);
```

从编译器的视角看，这个函数不过是：接收一个指向字符序列的指针，返回一个指向字符序列的指针——它甚至不知道"字符串"这个概念，只看到 `char*`。

于是你可以这样误用它：

```c
char *duplicate = strdup(some_string);
return duplicate;   // 函数在这里返回，没有释放 duplicate
```

编译器分析这段代码，会觉得一切正常：你传给 `strdup` 一个 `char*`，拿到一个 `char*`，又把它返回——类型完全匹配，挑不出毛病。

但是如果你读过 `strdup` 的 man page，会看到它写着：

> *"allocates memory for a new string with `malloc`; the memory should be freed with `free`."*
> （用 `malloc` 为新字符串分配内存；该内存应当用 `free` 释放。）

一个有经验的程序员看到上面那段代码就会马上意识到：`strdup` 返回的是一段**堆分配（heap allocated）**的字符串，但我们没有 `free` 就直接 `return` 了——这就是一处内存泄漏。

然而编译器看不到这些。编译器不读 man page，也不理解注释里那段自由英文。它**不知道这里存在一条后置条件**：「返回的是堆内存，必须由某人去释放」。

结论再强调一遍：你可以把前置条件和后置条件写进注释，但如果语言本身没有提供一种方式把它们**写进代码里**，编译器就无从得知你的意图。它能验证最基础的类型——你接收了一个数，你返回了一个布尔值，你返回了一个指向字符序列的指针——仅此而已。再往上一层的**意图**，它不懂，因此当你做错时，它也不会报错。

### 一个更好的类型系统？

这就带出本课的**核心问题**：

> **能不能设计一种更强的类型系统，让我们能把意图直接写在代码里，并由编译器去检查？**

如果编译器真的能理解你的前/后置条件——比如它能理解"这个函数返回一段堆分配内存，必须由某人调用 `free` 释放它"——那么它就可以反过来去检查：这块内存确实在某处被释放了吗？一旦能做到这一点，编译器就能替你核对你有没有遵守自己承诺的前置条件和后置条件，你的代码质量会因此大大提升。

**这正是 Rust 想要给我们的东西。**

接下来由 Julio 从更高层面介绍 Rust 是如何把"所有权"变成语言里的一等公民的。
## 进入 Rust：所有权模型

之前的讨论已经展示了"所有权"可以存在于注释或约定里，但语言本身并不会强迫你按约定走——没有什么机制能强制你释放内存。接下来要问的问题就是：**如果"所有权"这件事真的被写进编程语言里，会是什么样？** 下面先给出一个直观的 Rust 入门介绍，周四会再做深入讲解。

### 一行 Rust 代码与"谁拥有小熊"

看一下这一行 Rust：

```rust
let julio = bear_get();
```

形式上它跟你见过的其他语言差不多：声明一个变量 `julio`，让它等于某个函数的返回值。你可以想象 `bear_get()` 交给你的是一只小熊玩偶。

在 Rust 里需要记住的第一件事是：**每一个变量都"拥有"它持有的那个值**。这行代码里，`julio` 就是这只小熊的所有者（owner）——这只熊属于我，不属于别人，我可以对它做任何事。而既然我是它的所有者，**清理这只熊的责任也落在我身上**：玩完之后得把它放回去。

可以把场景想成一家日托所，架子上摆着一堆玩具。你从玩具架上拿一只玩具玩，玩爽了、玩够了，最后得把它放回你拿它的那个架子——这里对应的就是"玩具架"。

放到代码里就是这样一个场景：在 `main` 函数里写 `let julio = bear_get();`，当这个函数返回时，**是 `julio` 这个所有者——也就是"我"——负责把小熊放回去，没人会替我做**。

可以这样小结一下：等号左边的变量名（这里是 `julio`）是等号右边那个对象的所有者。我可以对这个玩具做任何事——按一下它身上的按钮让它发光，或者调用它的其他方法。但每次都要回到同一件事上：**把小熊放回去、释放它的内存，是所有者的责任。**

### 赋值触发所有权移动

给程序再加一行，情况就变得有意思了：

```rust
let julio = bear_get();
let ryan = julio;
```

一开始小熊在我（`julio`）手里。但当 `let ryan = julio;` 被执行，你应该在脑海里看到这样一幕：**小熊从 Julio 手里移动（move）到了 Ryan 手里**。

于是现在 Ryan 才是这只玩具的所有者。原来 Julio 能对小熊做的事——按按钮、让它亮起来、调用任何方法——**现在只有 Ryan 能做**。与此对应，**把小熊放回玩具架、释放这块内存的责任，也完全转到了 Ryan 身上**，和 Julio 没有关系了。

那问题来了：`julio` 这个变量现在会怎样？聊天区已经有同学答对了：**"Julio 什么都干不成了，这对我来说就结束了。"** 确实如此。从 `julio` 的视角看，小熊已经不在我这儿了——我把它交给 Ryan 了。我**没有所有权**，再去碰它只会出问题。在 Rust 里，任何尝试使用已经移出的 `julio` 的行为都会被编译器拦下。

不过也不全是坏消息：**我不再需要负责释放这只熊了**——不用把它放回架子上，也不用释放那块内存。这件事现在是 Ryan 的事，我可以安心走开。

你已经开始看出这个模型怎么回避之前讨论的那些内存问题了：现在"谁来释放"这件事被**显式地委托**给了某个唯一的所有者，一旦值交出去，责任就跟着走，不会有两个人同时以为该释放它。

### 一个看得见的例子

把上面的小熊换成一个字符串，跑一个真实的 Rust 例子：

```rust
let julio = String::from("hi friends");
let ryan = julio;
println!("{}", ryan);
```

右边可以是任何对象，这里具体是一个 `"hi friends"` 字符串。紧接着写 `let ryan = julio;`——和前面一样，这只"熊"（字符串）现在属于 Ryan 了。最后用 `println!` 打印（`println!` 加上感叹号，这是 Rust 打印的写法）。这段代码运行时一切正常：Ryan 是所有者，所以能打印出 `hi friends`。

但如果紧接着再试一次打印 `julio` 呢？

```rust
println!("{}", ryan);
println!("{}", julio); // 编译错误
```

如果换成别的语言，你可能以为这会输出两次 `hi friends`——毕竟 `julio` 和 `ryan` "看起来"指向同一个东西。但在 Rust 里，编译器能识别出：**这只小熊、这个字符串，已经不再属于 `julio` 这个变量了。** 编译器会给出一段信息量很大的报错，其中关键一句是：

```
error[E0382]: borrow of moved value: `julio`
```

翻译过来就是：`julio` 里原来持有的值已经被移到别处去了，而你又想再用这个变量——可是你手里已经什么都没有了，它不再归你。

换一种直观说法：`julio` 这个变量名"指向"它拥有的对象，而 `println!` 想做的是打印 `julio` 所拥有的东西。可到这一步 `julio` 什么都没拥有，所以只有 `ryan` 作为所有者时才能打印。

> **学生**：你是在说编译器那段 "borrow of moved value" 的报错措辞吗？
>
> **讲者**：是的。编译器其实没法"真的知道"你心里在想什么，这只是它在当前上下文里能给出的最准确的描述——"你原来放在 `julio` 里的那个东西，已经被移走了"。

换个比喻，你可以想象 **Rust 编译器就站在旁边、眼睛一直盯着：谁在拥有什么？什么时候被移走？** 一旦 `julio` 里的内容被移到 `ryan`，之后任何地方再出现 `julio` 这个变量名，编译器都会立刻反应过来："这个值已经不在了。" 正因为它盯得这么紧，才能给出很友好的报错信息，让这些问题更容易被发现。

### 把 `julio` 重新赋给别的东西

有同学问：那是不是 `julio` 从此就废了？并不是。**值被移走之后，你仍然可以把 `julio` 赋给另一个新的对象。** 比如现在 `ryan` 手里拿着那只熊，我可以再拿一只新的熊，让 `julio` 拥有它，这完全没问题——只要**每一个值都只有一个所有者**，这条规则不被破坏就行。

（聊天区问到："Rust 里变量不是默认不可变吗？" 是个很好的问题。关于不可变性和 `mut` 的规则，周四会专门讲，今天先从宏观上把所有权模型理解清楚。）

## 函数调用与所有权

上面看到的是**变量之间赋值**会触发所有权移动。还有一种常见情形也会触发同样的移动——**函数调用**。

```rust
let julio = bear_get();
my_cool_bear_function(julio);
```

这里把 `julio` 作为参数传给了 `my_cool_bear_function`。**这个函数就把小熊拿走了**——和前面 Ryan 从我手里拿走熊是一回事。于是调用之后：

- 在 `main` 里再试图用 `julio` 做任何事（比如打印）都会失败；
- **释放这只熊的责任现在转给了这个函数**，由它决定什么时候放回去；调用者 `main` 已经不再负责了。

## 借用：把分解找回来

这时马上会出现一个担忧：**那以后还怎么做代码分解（decomposition）？** Ryan 前面刚提过，CS 106A 头两周就在教分解；可现在又告诉你：把变量传进函数，就等于失去它的所有权，传完再也用不了——这几乎让任何想做的分解都没法做。

好在 Rust 还有另一个特性叫**借用（borrowing）**。直观想一下：我持有这个玩具，我愿意把它**借给**别人用一小会儿——让他们打印它一下，或者做点别的——然后他们**用完得还给我**，因为它本来就是我的。

写成代码就是加一个 `&`：

```rust
let julio = bear_get();
my_cool_bear_function(&julio);
```

如果你有 C 背景，看到 `&`（ampersand）可能会犯嘀咕："我是在传地址吗？" 从底层看它确实是引用。但更贴切的理解是：**我允许这个函数借用我的小熊、借用我的字符串、借用我的对象；函数返回之后，这只熊还在我手里**，我可以继续像以前一样写代码，根本不用担心变量"被移走了"。

简而言之：**借用出去了要记得还，释放的责任仍然在原来的所有者身上，只是有那么一小会儿这个东西在别人手上。**

讲者举了个更生活化的类比：**就像我把任天堂 DS 借给朋友玩**——理论上他会还给我，然后 DS 还在我这儿。Rust 里的借用就是这样。

有了 `&` 这个借用机制，**你熟悉和喜欢的那种代码分解就回来了**：函数可以操作值，但不必吃掉值。

## 小结

把上面的内容拉到宏观层面，Rust 官方教程（Rust Book，课件里有链接，写得非常清楚）对所有权的总结基本就是这样几条：

- **每一个值，在任意时刻只能有一个所有者。**
- **当这个所有者离开作用域时，值会被释放。** 比如前面 `main` 函数走到右花括号 `}` 的那一刻，Rust 就会替你把那时该释放的东西释放掉。你可以想象所有释放动作都是在这些花括号处、由编译器按"谁是所有者"这条线索自动安排好的。
- **谁是所有者，谁就是释放的责任人**——这正好避免了前面讨论的那种双重释放（double free），因为不会再出现两边都以为自己该释放的情形。
- **最关键的一点：所有这些检查都发生在编译期（compile time）。** 程序根本没跑起来，就像刚才那个字符串例子一样——编译器直接拦住你："你违反了所有权规则，我不知道你想做什么。" 这就是 Rust 相比 C 的最大优势：C 可以让你在没有释放的情况下照样编译通过，因为这类检查根本没有内建在 C 编译器里；而 Rust 把这些检查内建进编译器，对内存安全问题帮助非常大。

下一次课我们会再讲 Rust 里其他类型的变量、这套系统还能回避哪些内存问题，并看更多代码示例。

### 课堂尾声问答

> **学生**（下课离开前的一个问题）：在 Rust 里，栈（stack）和堆（heap）还存在吗？
>
> **讲者**：当然还在。周四我们会更细地讲内存布局，但你可以这样想：刚才例子里的那只"熊"、或者那个字符串，是**在堆上**的；而等号左边的 `julio`、`ryan` 这些变量本身**在栈上**，相当于指向堆上对象的指针——和 C 的模型没什么两样，只是在这个模型之上多加了一层由编译器强制执行的规则。

所有权一开始会让人觉得有点别扭，头几次你肯定会跟编译器"打一阵架"，但一旦这个心智模型建立起来，后面的一切都会顺很多——因为它真正把"这块内存到底该由谁来释放"这件事说清楚了。
